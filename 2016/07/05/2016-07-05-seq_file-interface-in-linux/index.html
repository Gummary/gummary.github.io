Linux中的proc伪文件系统和seq_file接口

发表于 2016-07-05 | 分类于 Linux |

今天的操作系统课程实践是让基于proc写一个读取进程控制块信息的内核模块，开始以为这两个东西是捆绑到一起的，后来在网上查阅了一些资料发现这是两个东西。这篇博客先介绍一下什么是proc伪文件系统，然后再用一个简单的例子来说明一下seq_file接口。这两部分内容大多都是翻译自一些英文文章，和之前一样也会加入一些自己的理解。

proc伪文件系统

在Linux中proc伪文件系统主要是用于内核和内核模块给应用进程发送信息，一开始是用与提供给应用程序一种简单的方法来获取进程的信息（这也是叫proc的原因），后来被用于内核为用户进程提供各种信息。

和上一篇博文讲的设备差不多，使用proc文件系统，也就是创建一个proc文件必须要创建一个结构体，其中包含了指向操作该proc文件的指针，还有提供给init_module和cleanup_module注册和卸载模块的函数。

proc文件系统并不是存放在硬盘上而是存放在内存中。

在proc文件被打开或关闭时，模块编写者并不能得到通知，因此可能会出先proc文件被打开后，该模块被移除了的情况。

seq_file接口

创建一个正确的proc文件是非常困难的，当输出边的特别大的时候，读取速度会非常慢，而且会出现一些错误，但是在linux2.6之后的内核中，包含了一系列的函数，用于快速简单的读写proc文件，这些函数就是seq_file接口。

sqe_file接口包含在<linux/sqe_file.h>文件下，sqe_file包括了三个方面：

  1. 一个迭代器接口来逐步的获取文件内容。
  2. 一些函数来格式化输出而不用担心输出的内存缓冲的大小等信息。
  3. 将操作虚拟文件的函数封装成几个文件操作函数。

迭代器接口

使用seq_file接口来实现一个虚拟文件的内核模块必须要实现一个简单的迭代器。这个迭代器可以逐步的读取数据。迭代器要有能力移动到一些和文件实现相关的特殊的位置，但对这些特殊位置的解释又取决于迭代器自身。假设用seq_file来实现读取防火墙规则，那么迭代器就可以把位置n解释成第n条规则。因此位置取决于文件是如何实现的，而不必去计算在文件中的偏移量。唯一的例外就是位置0代表着文件的开始。

/proc/sequence的迭代器是将下一个要输出的数字作为位置。

要使迭代器工作必须要有4个函数，分别为start，next，stop和show，下面将依次介绍这四个函数。

stop

stop函数是在迭代完成时被调用，作用是清理，如果在迭代的过程中申请了内存，那么就要在stop中释放掉。

show

show函数会格式化迭代器所指向的数据并将其输出。

如果一切做的都正确的话，show函数会返回0，而一个负数意味着出现了错误，错误会返回到用户空间。这个函数也可以返回SEQ_SKIP，表示跳过当前值，如果在这之前show已经产生了一些输出，那么这些输出将会被遗弃。

最后通过定义一个seq_operations结构体来完成对seq_file的定义。seq_operations中包含了那4个我们刚刚定义的函数。

值得注意的是由start返回和由其它函数操作的迭代器的内容对于seq_file接口来说是完全不透明的。 It can thus be anything that is useful in stepping through the data to be output. Counters can be useful, but it could also be a direct pointer into an array or linked list. Anything goes, as long as the programmer is aware that things can happen between calls to the iterator function.seq_file在start和stop之间不会进入sleep状态。
