Automake的介绍

在构建大型的C或C++项目时，很难编写出一些复合GNU Makefile标准的Makefile的文档，所以就产生了一些工具来自动生成Makefile文件。如果下载过一些C或C++项目的源码的话，可以看到，在目录下有很多configure, configure.ac, Makefile.in, Makefile.am, aclocal.m4类似的文件，这些文件的作用就是用于生成最终的Makefile。

GNU 构建系统简介

在Unix世界中，通常用make构建一个系统，make的功能是读取package中的Makefile文件。Makefile包含了一系列用于构建系统的规则。每次make运行时都会读取Makefile然后检查文件是否存在是否被修改等信息，根据这些信息决定那些文件需要被构建（或重构），然后运行与之相关的命令。

当一个系统需要在一个与开发环境不同的平台构建时，Makefile文件通常需要调整，例如编译器的名字变了或者需要其他的参数。在1991年David J. MacKenzie受够了为20个平台定制Makefile，于是他写了一个叫configure的shell脚本来自动调整Makefile（见Genesis ）.编译他的系统只需要简单的运行./configure && make即可。

如今这个过程被GNU项目标准化了了，GNU编码标准(见The Release Process) 规定了每一个GNU项目的configure文件的内容和最小的接口。Makefile文件也应该遵循一些已经建立的标准。这么做的结果就是一个统一的构建系统可以使所有的系统在安装时都相同。

这个构建系统就称作 GNU Build System.因为它就是通过GNU项目发展起来的。

Autotools就是为你的工程构建一个GNU Build System的工具。Autoconf用于生成configure脚本，Automake用于生成Makefile文件。不使用这些工具也可以构建GNU Build System，但是非常的复杂而且易于出错。

2.2.8 交叉编译

交叉编译是在一个平台上构建另一个平台上运行的程序.在交叉编译中，区分出构建平台和运行平台。构建平台是执行编译任务的平台，而宿主平台是最终的二进制文件运行的平台。下面两条configure选项就是用于指定这了个平台的。

--build=build

这个指定了构建系统的平台

--host=host

这个指定了宿主平台

当--host被指定后，configure会搜索相应的交叉编译套件。通常交叉编译工具都会用他们的目标平台作为名称的前缀。例如，MinGW32的交叉编译工具的名称是i586-mingw32msvc-gcc, i586-mingw32msvc-ld, i586-mingw32msvc-as等等

--target=target

当编译编译器时，指定编译器对应的目标平台

例如当安装GCC时，我们可以使用 --target=target来指定我们所构建的GCC是作为target平台的交叉编译器。混合使用--build和--target,就可以交叉编译一个交叉编译工具。这种类似的交叉编译方式通常称为Canadian cross

2.2.11 准备发布

我们曾经提到过make dist。这个命令是收集所有的源文件和构建系统所必须的源文件来创建一个tar包，tar包的名字为package-version.tar.gz。

另外一个更有用的命令式make distcheck,distcheck目标构造的tar包和dist一样，但是这个命令额外的确保了至今所说的命令都可以工作。

  • 这个命令尝试了一个完整编译过程，解压新构建的tar包，运行make，make check， make install，make installcheck甚至make dist。
  • 尝试在只读目录下VPATH 构建系统
  • 确保make clean, make distclean, make uninstall没有忽略任何一个文件。
  • 接茬DESTDIR安装工作。

所有的这些工作都在一个临时的目录下进行所以不需要root权限。但是也要把真实的子目录结构作为一个实现的细节考虑，因为这是随时可以变的。不要全部都依赖于这个命令。

如果运行make distcheck失败了，那就说明上述的一些操作无法正确的执行。一些用户可能会很沮丧。因此最好是成功运行make distcheck之后再发布程序。这个教程并不能保证构建的系统完美无瑕，但是能防止一些在安装时出现的不必要的错误。

2.2.12 自动依赖追踪

自动依赖追踪是编译的一个副作用。每当构建系统编译源文件时，都会计算一个依赖库的列表。然后每当运行make并且依赖的文件被改变了时，依赖文件都会被重构。

如果开发者不覆盖配置，默认情况下Automake会生成代码来自动进行依赖追踪（见Dependencies ）。

在执行configure时，你可以看到他在检查每一个它支持的依赖机制

因为依赖仅仅被算作编译时的一个副作用，所以在构建一个包时没有以来信息会被记录。这是完全可以的是因为所有的文件都需要被构建一次，make并不需要知道那些文件需要被重构。实际上，依赖追踪对于一些只构建一次的包完全没有用，所以，configure有一个选项来禁止。

--disable-dependency-tracking

加速只构建一次的软件包

有一些编译器并没有提供有效的方法来派生出以来的列表，此时就需要单独运行一些程序来完成这项工作。由于单独运行非常的消耗性能，所以默认是禁止的。可以使用--enable-dependency-tracking选项来激活它们。

--enable-dependency-tracking

不禁止慢速的依赖提取。

嵌套包

虽然嵌套包并不推荐给想使用Autotools的人，但是在这个教程里还是有必要提及的。

使用Autotool是创建的包可以被嵌套到任意的深度中。

通常的步骤是包A把他需要的依赖库放到一个独立的子目录下，而依赖库B有着自己的GNU构建系统。那么A的configure脚本在运行时也会运行B的configure脚本，构建和安装A也会构建和安装B，生成发行版也会包含B。

很有可能一个包下包含多个这种包。GCC就是使用了这种特性。这给了安装着一个独立的包configure，build和install，同时也允许开发这独立的工作在子目录下。

当构建嵌套包时，传递给顶层configure的选项也会递归的传递给嵌套的包，如果一个包不能解析一个选项，就会忽略它，并假设对其他包有效。

命令configure --help=recursive可以用于显示包括的包所支持的选项。

Autotools是如何提供帮助的

下面是一些你可能不想亲自写GNU构建系统的原因。

  • 正如上面所写的，GNU构建系统有许多的特性，可能有一些用户需要的特性你没有完成因为你并不需要他们。
  • 实现这些特性可能非常困难而且很烦人。想象一下，你要书写正确的shell脚本，正确的Makefile，并且可能其中一些是为你不熟悉的系统架构编写的。
  • 你将会升级你的步骤来保证与GNU编码标准一致。

GNU Autotools会隐藏所有的这些细节并提供

  • 工具来产生合适完整自包含的GNU构建工具。自包含意味着构建这个系统不会要求GNU工具。
  • 一个提供完善和提高的中心场所。

但是也有一些原因你可能不会使用Autools，例如你可能已经使用了其他的构建系统，Autotools只有在你接受GNU构建系统的情况下才有用。

autoconf（生成configure脚本）和automake（生成Makefile.in）都会读取这个文件。它包含了许多M4宏，这些宏最终都会被扩展成configure脚本的的shell代码。这里不会说明如何编写这个脚本，在Autoconf手册里有详细的一节来介绍(见Writing-Autoconf-Input ）

configure.ac中以AC_ 为前缀的是Autoconf宏，在Autoconf手册中有详细的介绍(见Autoconf-Macro-Index )。AM_为前缀的是Automake宏，在Automake文档中有详细的介绍(见Macro-Index )

AM_INIT_AUTOMAKE中的参数是一系列传递给automake的选项。-Wall和-Werror是令automake打开所有的Warning并将他们报告为error。这里我们讨论的是automake的Warning，例如Makefile.am中未知的指令。这个Warning与编译器的Warning之间没有任何关系，即使二者可能有着相同的名称。在编写一个软件包时使用这两个选项可以帮助你不遗漏任何的问题，这会让你以后很轻松。foreign选项告诉automake这个包不会与GNU标准完全一致。GNU包应该在发布时必须包含额外的文件例如ChangeLog，AUTHORS等等。在我们这个简单的例子中没有必要加入这些。

AC_PROG_CC这一行会使configure脚本搜索一个C编译器并为CC赋值。由Automake生成的src/Makefile.in会使用CC来构建hello。所以当configure根据src/Makefile.in创建src/Makefile时，会用他找到的编译器来定义Makefile中的CC。如果要求Automake在Makefile.in中使用一个configure.ac未定义的CC时，他会建议你增加AC_PROG_CC的调用。

AC_CONFIG_HEADERS([config.h])调用会收集由所有位于configure.ac中的其他宏定义的"#define"内容，然后存到config.h文件。在这个例子中，AC_INIT已经定义了许多宏。下面是在configure运行之后config.h的部分内容。

你可能会注意到在src/main中也使用了config.h头文件，hello程序就可以使用PACKAGE_STRING宏了。在一个真实的软件包中，config.h会变得非常巨大，每一个#define都包含了系统的一个特性。

AC_CONFIG_FILES宏声明了所有由configure根据*.in模板生成的文件。Automake也会遍历这个目录来找到他需要运行的Makefile.am文件。（很重要的一点是你的工程每添加一个目录，你就要添加一个Makefile到这个列表中，否则Automake永远不会运行你文件夹下的Makefile.am文件）

最后AC_OUTPUT是一个结束命令，这个命令生成的脚本代码的作用是生成由AC_CONFIG_HEADERS和AC_CONFIG_FILES注册的文件。

当开始一个新的包时，我们建议你通过写一个简单的 configure.ac文件开始，然后再逐渐增加一些其他的测试。autoscan命令也会建议一些你的包可能需要的测试(见autoscan-Invocation )

2.4.3 amhello的Makefile.am解释

现在我们来看一下src/Makefile.am文件。这个文件包含了Automake用于构建和安装hello的指令。

Makefile.am和Makefile文件有着相同的语法规则。当automake加工Makefile.am是将整个文件复制到Makefile.in文件(会被configure转换为Makefile)中。会通过生成一些构建规则和一些其他的变量来反应些变量的定义。通常Makefile.am文件中只包含了上面列出的那些变量。但是也可以包含一些automake不会进行解释的其他的变量和规则的定义。

以_PROGRAMS结尾的变量列出了最终Makefile应该生成的程序名称。从Automake的角度讲，_PROGRAMS后缀称作主语。Automake会将其他主语例如 _SCRIPTS, _DATA, _LIBRARIES等与对应的文件相关联。

bin_PROGRAMS的bin部分会指定automake将程序安装到bindir。GNU构建系统有一系列的变量来表示目的目录并允许用户定制这些路径。任何一个这种类似的路径变量都可以作为前缀放在一个主语的前面（除了dir后缀）。以此来告诉automake将程序安装到哪。

程序需要从源代码构建，所以每一个在_PROGRAMS中被列出的程序prog，automake会寻找另一个叫做prog_SOURCES变量，这个变量列出他的源代码。可能会有一个或多个源代码，他们会被一起编译和连接。

automake同样知道当创建一个tar包来发布程序时，源代码也要被打包。所以hello_SOURCES变量的另一个作用是在使用make dist创建tar包时main.c也会是tar包的一部分。

最后是最高层的Makefile.am的解释

SUBDIRS列出了所有在构建当前目录之前make应该先进入的目录。所以当我们在最高的目录结构上运行make时，make会进入src目录的原因就是定义了SUBDIRS变量。

dist_doc_DATA变量使得README在发布和安装时被安装到docdir。由_DATA为后缀的变量列出的文件不会在发布时自动加入tar包，所以加上dist_前缀使他们也能发布。然但对README来说是多此一举的，因为automake会自动的发布他所遇到的所有README文件(其他由automake自动发布的文件可以通过automake --help)查看。第二行唯一重要的作用就是在make install时安装REAME

这个例子中没有包含的就是在程序中如何获取安装目录的值(见Standard-Directory-Variables )

也就是如何把他们转换成宏变量。详细内容可查看(https://www.gnu.org/software/autoconf/manual/autoconf.html#Defining-Directories )
